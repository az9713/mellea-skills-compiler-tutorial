# Extension Guide

The Mellea Skills Compiler is designed with explicit extension points. This page documents what can be extended, what can be built on top, and the intended interfaces for each.

---

## Extension point 1: Skill specifications

The simplest extension is adding new skill specs to `skills/`. No code changes needed.

```bash
mkdir skills/my-new-skill
# write skills/my-new-skill/spec.md
mellea-skills compile skills/my-new-skill/spec.md
```

The 16 skills in `skills/` are the current library. Contributions welcome — see `CONTRIBUTING.md`.

**What makes a good addition to the skills library**:
- A distinct archetype (A, B, C, D1, D2, or E) not yet covered
- A real-world use case with concrete tool calls or domain knowledge
- T1 or T2 tier (avoids requiring specific credentials or services to run)

## Extension point 2: New export targets

The `mellea-skills export` command currently supports `langgraph`, `claude-code`, and `mcp`. New targets are added by implementing an adapter in `src/mellea_skills_compiler/export/adapters/`.

Each adapter implements the `Adapter` protocol (from `export/exporter.py`):

```python
class Adapter(Protocol):
    def translate(self, ctx: LoadedContext) -> TranslationPlan: ...
    def emit(self, plan: TranslationPlan, out_path: Path) -> int: ...
```

`LoadedContext` contains:
- `manifest` — the parsed `melleafy.json`
- `package_path` — path to the compiled package
- `pipeline_fn` — the importable `run_pipeline` function

`TranslationPlan` contains:
- `files` — list of `(relative_path, content)` tuples to write
- `bundle_package` — whether to copy the compiled package into the output

The adapter receives a fully parsed, introspected compiled package and is responsible only for generating the target-specific wrapper files.

**Adding an adapter for a new harness**:

1. Create `src/mellea_skills_compiler/export/adapters/<harness>.py`
2. Implement the `Adapter` protocol
3. Register the target in `export/exporter.py:_TARGET_REGISTRY`
4. Add tests in `tests/mellea_skills_compiler/export/`

The exporter handles the validate, load, emit, and lint stages; the adapter handles only translate and emit.

## Extension point 3: New dialect docs

The compiler handles source specs from multiple frameworks (CrewAI, LangGraph, Letta, etc.) via dialect documents in `docs/dialects/`. New dialects are added by creating a new dialect doc and updating the Step 0 classifier.

A dialect doc (see `docs/dialects/agent-skills-std.md` as the reference implementation) specifies:
- Which files to read in Step 1a (inventory)
- How to interpret them in Step 1b (element tagging)
- Overrides for the general Step 2 mapping table

Three dialects currently lack full dialect docs (`autogen`, `openai_agents_sdk`, `smolagents`) — contributions here would improve compiler quality for those frameworks.

## Extension point 4: Additional Guardian risk checks

The certification pipeline reads risk checks from the `PolicyManifest` generated by Nexus. The manifest specifies which Guardian risk categories to enable, with system prompts.

To add a custom risk check:
1. Add a new entry to the `PolicyManifest` (or extend Nexus to emit it for your use case)
2. Configure the Guardian hook with the custom system prompt and risk category
3. Guardian evaluates it against every `m.instruct()` output

The Mellea hook system supports custom plugins beyond Guardian. See the Mellea library documentation for the plugin API.

---

## Extension point 5: Compliance mappings

The AUTOMATED / PARTIAL / MANUAL classification uses YAML files under `src/mellea_skills_compiler/certification/data/`. These map governance action IDs (from NIST AI RMF and Credo UCF) to pipeline control identifiers.

To extend compliance coverage:
- Add new governance frameworks by creating a new taxonomy mapping file
- Add new pipeline controls (e.g., "has type validator", "has audit trail") by extending the control inventory
- Update the classification YAML to map new action IDs to the new controls

The classification system is intentionally static YAML rather than LLM-driven — this makes it auditable and deterministic. Ground-truth validation of these mappings against human expert assessment is planned future work.

## What can be built on top

### Skill registries

A registry of pre-compiled skills (`<name>_mellea/` directories) that users can pull and run without compiling. The `melleafy.json` manifest provides the stable interface for a registry entry: package name, entry signature, modality, parameters, and env var requirements.

A registry could be as simple as a directory of compiled packages or as complex as a versioned artifact repository with search, tier filtering, and certification status.

### Automated regression testing

Run all fixtures for all compiled skills on every commit:

```bash
for dir in examples/*/; do
    mellea-skills validate "$dir"*_mellea/ --all
done
```

The structured exit codes (0, 11, 12) and JSONL output make this scriptable. Guardian verdicts from the runs accumulate in `audit_trail.jsonl` — useful for detecting model regressions when the underlying LLM changes.

### Per-phase model routing

Decomposed pipelines expose a natural optimization: each Mellea session in `pipeline.py` is a separate model invocation. Classification and extraction phases (archetype D1, early phases of A/C) tend to work well with smaller models; complex reasoning phases need larger ones.

Today, all sessions in a compiled pipeline use the same `BACKEND` and `MODEL_ID` from `config.py`. A per-phase routing extension would:
1. Tag each session in `pipeline.py` with the reasoning type (extract, classify, generate, reason)
2. Look up the appropriate model for that type from a routing table
3. Override `BACKEND` and `MODEL_ID` per session

This is directly buildable on top of the compiled package — `config.py` is editable and `pipeline.py` already uses `BACKEND` and `MODEL_ID` as variables.

### Specification quality tooling

An incidental benefit of compilation is that decomposition surfaces spec quality issues. A standalone spec linter would:
1. Run Steps 0–2.5 (classification, inventory, mapping, dependency audit)
2. Stop before code generation
3. Report: contradictions in constraints, underspecified tools (C6 abstract without detail), output types without enough schema structure, elements that compile to stubs because the spec is vague

This is implementable today by running `mellea-skills compile --no-run` and inspecting `intermediate/dependency_plan.json` and `mapping_report.md` without generating any Python code.

### Ecosystem-scale governance

Applying certification to every skill in a registry — running `mellea-skills certify` on a large collection of compiled skills and aggregating the `audit_trail.jsonl` and `CERTIFICATION.md` outputs into a governance dashboard. The structured artifacts make this directly scriptable.

## Stable vs. unstable interfaces for extension

| Interface | Stability | Safe to build on |
|-----------|-----------|-----------------|
| `run_pipeline` entry point | Stable | Yes |
| `melleafy.json` (≥ 1.0.0) | Stable | Yes |
| `fixtures/` contract | Stable | Yes |
| `Adapter` protocol in `export/exporter.py` | Unstable (experimental) | With caution |
| `intermediate/` JSON shapes | Unstable | No |
| CLI exit codes (0, 11, 12) | Stable | Yes |
| `audit_trail.jsonl` format | Stable | Yes |
| `policy_manifest.json` format | Stable | Yes |
| Python API in `export/exporter.py` | Unstable | No — use CLI |
